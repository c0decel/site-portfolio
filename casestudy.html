Overview

Transparent Storefront is a commerce template developed with the MERN stack that allows users to display financial data and production to their customers

Purpose and context

I created this template as a resource for business owners to publically display records and details that customers and clients don't normally see. There has been an increase of ethical consumers who are conscious about how and where they spend their money, and small businesses can use this to their advantage by offering transparency of company operations in order to built trust with customers.

Objective

The goals were to make a free template with unique for small business owners that is easy to set up and use, and to have a detailed full stack project to display on my portfolio.

Process

First I created a new Angular project and set up a few basic pages and functions, and made a mock API to test them out on. I did this to get a good idea of how to structure the actual API.

Next I created a MongoDB database, my models, and some basic features such as creating users, products, and tags.

Next I added some security features like CORS, user authentication, and admin permissions and tested my endpoints. Once everything worked, I deployed my API to Heroku.

Finally I replaced the mock data on the front end with my live API and fully fleshed out the placeholder functions and set up admin powers.

Successes

I think that this project improved my ability to narrow down where errors originate from and fix them quickly, which is something I struggled with before that led to me taking longer than neccessary to get past errors.

It also gave me a better understanding of how Angular works, and how to test and determine if errors originate from the front end or back end.

Challenges

Most of the challenges I came across related to making sure my front end was compatible with my endpoints. After getting ahold of it it became much easier to find where errors were originating.

Another challenge I had was linking different parts of different models together, but doing so gave me a better understanding of how to best pass information around the database as well as structure it for more complex and frequently changing data.

This project took longer than expected due to the size, and a few features were harder to implement than initially thought.

I found that adding features in "steps" helps to keep me on track and organized, and it also requires less going back and modifying things to make them accomodate different features or changes.

Future steps

I plan to continue adding features and tools such as supply inventory, batch numbers, and a rewards system.

Thoughts

I think this project was worth the time put into it, and it displays my understanding of both front end and back end development.
